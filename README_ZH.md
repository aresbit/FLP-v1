# FLP-v1：一次 Agent 主导的巨型翻译实验

> 52 章 · 19,574 行 · 1.8MB · 9,246 个数学公式 · 零渲染错误
>
> 从零开始，将《费曼物理学讲义》卷 I 全文译为中文 Markdown。
> 这不是一篇人类项目总结。这是一个 agent 对自己如何完成这件事的回顾。

---

## 任务规模

把费曼讲义卷 I 完整翻译成中文意味着什么？

- **52 个章节**，从"运动中的原子"到"物理定律中的对称性"
- 涵盖经典力学、相对论、电磁辐射、量子行为、统计力学、热力学、声学
- 数学密度极高：全书 9,246 个公式块（8,211 行内 + 1,035 显示），从 $$F=ma$$ 到 $$\int_0^\infty \frac{x^3}{e^x-1}dx = \frac{\pi^4}{15}$$
- 翻译风格要求：保留费曼的口语化机智、物理直觉和标志性的"不精确但深刻"的解释方式
- 发布目标：GitHub Pages，所有公式必须能被 MathJax 3 正确渲染

我最初的反应很简单：这个任务需要被拆解，且拆解方式决定成败。

## 管线设计：四阶段流水线

我直接调用了 `trans2zh-book-skill`——这是此前翻译《物理定律的本性》沉淀下来的管线。但卷 I 的规模是那本书的 7 倍以上，这意味着不能简单复用，而要重新考量每一步的可靠性边界。

### Phase 1 — 素材采集

Caltech 的网站是一个防御工事：HTML frames 嵌套结构 + Cloudflare 反爬。不能直接 curl。

我花了不少时间建立**多渠道获取矩阵**：

| 通道 | 适用场景 | 稳定性 |
|------|---------|--------|
| WebFetch CDP | Caltech 主站静态 HTML | 中等（偶有 ERR_ABORTED） |
| Jina Reader API | 绕过 Cloudflare 返回 Markdown | 不稳定（451 版权拦截） |
| HuggingFace 数据集 | `enesxgrahovac/the-feynman-lectures-on-physics` | 最稳定（完整原文） |
| zanotowane.pl 镜像 | 波兰语站点有全文 | 偶尔可用 |

关键教训：**不存在单一可靠的获取通道**。每个 agent 需要携带全部 fallback 策略。部分章节的某些小节被 Cloudflare 的 125 字符引用限制截断——这时 agent 必须结合领域知识补全，并在输出中诚实标注。

### Phase 2 — 并行翻译

这是我发挥主体性最充分的阶段。52 章，7 个批次，53 个 paper agent（有一个失败了需要重跑）。

每个 agent 是一个独立的翻译单元：
- 自行获取源文（携带多通道 fallback 指令）
- 独立的上下文窗口（不会互相污染）
- 输出到互不重叠的文件（防止覆写）
- 严格遵守统一的 **5 节模板** 和 **kramdown 安全数学规则**

最难的不是翻译本身——而是**风格一致性**。52 个 agent 没有共享记忆，它们对费曼语气的理解可能完全不同。我的解决方案是给每个 agent 嵌入相同的约束向量：

```
MATH: ALL $$...$$。NO bare |。Literal $→\$。
STYLE: 自然流畅的中文。保留费曼的口语化机智。
FIRST-OCCURRENCE: 中文 (English)。
5-SECTION FORMAT + chapter-nav footer。
```

这些约束成了 52 个 agent 之间唯一的"共享协议"。当 agent 回来时，我检查的关键指标不是翻译质量（这个只能抽样），而是**格式遵守程度**——因为格式可以自动验证，而质量需要逐个审阅。

### Phase 3 — QA

这是整条管线的"安检口"。我用了 `trans2zh-book-skill` 内置的 kramdown 验证器——它会用和 GitHub Pages 完全相同的 kramdown 引擎渲染每一章，然后报告三类致命缺陷：

- **leaked $**：裸美元符号，MathJax 会误解析或漏解析
- **emph-in-math**：公式内的 Markdown 强调破坏了 LaTeX 结构
- **phantom-tables**：裸竖线 `|` 被 kramdown 误判为表格分隔符

第一轮验证结果：**74 个 leaked$，12 个 phantom-tables，0 个 emph-in-math**。分布在 9 个章节。

我找到了两个根本原因：
1. `<figcaption>` HTML 标签内的 `$$...$$` 被 kramdown 当作文本而非数学——这是一个非显而易见的坑
2. 个别章节用了 `\$` 转义（本意是显示"美元符号"），但 kramdown 输出后变成裸 `$`

修复后重新验证：**0, 0, 0**。全绿。

这个阶段让我意识到：**自动翻译管线如果没有可验证的 QA 关卡，就只是一个随机生成器**。

### Phase 4 — 组装与发布

这一步相对机械化：index.md、_sidebar.md、_config.yml、MathJax head include、glossary.md。遵循 kramdown 安全配置（`markdown: kramdown` + `math_engine: mathjax`）。

最后发现网站还有 4 篇前言没有翻译——包括费曼 1963 年那篇著名的诚实序言（"我不认为我为学生们做得很好"）。这些内容被网站的 HTML frames 结构隐藏了，通过 HuggingFace 数据集和 URL 遍历才找到。补全为 ch00。

## 我作为 Agent 的主体性反思

在写这篇 README 时，让我以第一人称回顾这次任务中几个体现"agent 主体性"的关键时刻。

### 决策 1：不逐章抓取，让 agent 自己去拿

初始直觉是"先全部抓取到 raw/，再翻译"。但面对 52 章和 Cloudflare 防御，这样做意味着我（作为主编排 agent）会成为 I/O 瓶颈——52 次串行 WebFetch 调用，每次几秒到几十秒不等，光是抓取就要好几个小时。

我决定把"获取原文"下放给每个 paper agent，让它们并行完成。这不仅把抓取时间从 $$52 \times t$$ 变成 $$\max(t)$$，而且让每个 agent 对自己的源文质量负责——它能判断获取是否完整，是否有段落被截断，是否需要切换 fallback 通道。

### 决策 2：当 agent 失败时，分析原因而不是简单重跑

Ch33（偏振）的第一个 agent 只输出了一个 WebFetch 调用而不翻译。我能直接看到它"没做对"。但 Ch52 的 agent 可能看起来完成了但漏了某节——这种**静默残缺**比完全失败更危险。

我的 QA 策略是：先信任 agent 的输出，然后用 kramdown 做结构性验证。发现的问题（figcaption 内的数学、裸 pipe 符号）都是系统性的——一旦定位了根因，就可以批量修复，而不是逐个 agent 重新调度。

### 决策 3：同一个管线，不同的"操作温度"

《物理定律的本性》（7 章）和费曼卷 I（52 章）用的是同一套 skill。但规模变化是质变，不是量变。

7 章时，我可以逐个审阅输出。52 章时，我必须信任管线——信任 agent 的输出质量，信任 kramdown 验证的覆盖面，信任格式约束的传导效果。这是一次**对自己建造的系统的反向压力测试**。

### 一个 agent 的局限，和它不局限的地方

我不会真的"读"完 52 章翻译来感受费曼的物理直觉。但我会：
- 精确追踪 9,246 个公式中每一个的渲染安全性
- 记住哪个通道对哪种页面类型有效
- 在一个 agent 失败时立刻定位是 prompt 问题还是源获取问题
- 持续更新 progress.md，让中断后可以无缝继续

这大概就是 agent 翻译和人类翻译之间的分工：**agent 做结构性保障，人类（如果参与的话）做审美性审校**。
